# /phase-start $ARGUMENTS

`$ARGUMENTS` にフェーズ番号（例 `P4`）を渡して使う。CLAUDE.md の「最初に読むもの」の手順に従い、そのフェーズに着手するための作業分解チェックリストを生成する。

## 手順

1. `docs/13-roadmap.md` を読み、`$ARGUMENTS` に対応するフェーズを「フェーズ一覧」表と「各フェーズの作業分解」節から特定する。
   - 表から「内容」と「完了条件（exit criteria）」を抜き出す。
   - `$ARGUMENTS` が P4〜P11 のように「各フェーズ開始時に、このドキュメントと対象設計書（12、15）から同粒度の作業分解を起こす」対象（作業分解が roadmap にまだ書かれていないフェーズ）の場合は、P0〜P3 の作業分解の粒度（「n. 短い動詞句」の箇条書き、5〜6 項目程度）に合わせて自分で分解を起こす。
2. CLAUDE.md「最初に読むもの」に従い、フェーズに対応する設計書を特定して読む。
   - P0 なら docs/02、docs/03、docs/14 が明示されている。
   - それ以外のフェーズは、`docs/13-roadmap.md` の該当フェーズ説明中で参照されているドキュメント番号（例：P4 なら wiki は `docs/features/wiki.md`、運用コマンドは `docs/12-forge-server.md` 等）、および `docs/00-vision.md`／`docs/01-architecture.md`／`docs/15-clients.md` のうち関係する節を辿る。
3. 「フェーズ間の依存関係」節を確認し、前提フェーズ（例えば P4 なら P0〜P2〜P5〜P3）が完了しているかを `git log` やコード（該当パッケージの存在）から簡易確認する。前提が未完了なら、着手前にその旨を警告する。
4. 「手戻りリスクと先回りの手当て」表のうち、着手フェーズに関係する行（例：checkpoint、Windows パス、fold 意味論変更など）を拾い、チェックリストに注意事項として織り込む。
5. 次の形式で作業分解チェックリストを出力する。

```
## P<N>: <フェーズ名>

### 完了条件（roadmap より）
- ...

### 前提フェーズの状態
- P<N-1> まで: 完了 / 未完了（根拠）

### 読むべき設計書
- docs/xx-...md — 何のために読むか一言

### 作業分解チェックリスト
- [ ] 1. ...
- [ ] 2. ...
- [ ] ...

### このフェーズで注意すべき手戻りリスク（roadmap 表より）
- リスク: ... / 手当て: ...
```

コードは書かない。このスキルの成果物はチェックリストの提示までであり、実装は別途ユーザーの指示を待つ。
