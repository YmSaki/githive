# 00. ビジョンと原則

## 解決する問題

プロジェクトの記憶（issue、タスク、議論、通知、メンバー情報、wiki）は、現状では GitHub や GitLab といったホスティングサービスのデータベースに閉じている。
リポジトリを clone してもコードしか手に入らず、文脈は API 経由でしか取得できない。

この構造は AI Agent にとって特に不利に働く。
Agent は clone 直後にプロジェクトの文脈を持たず、文脈を得るには各サービス固有の API、認証、レート制限に個別対応する必要がある。
人間にとっても、オフラインでは issue が読めない、サービスを移行すると履歴が失われる、という制約になる。

githive はこの記憶を git そのものに移す。
記憶は refs 配下のコミット履歴とテキストファイルとして保存され、clone（と fetch 設定一行）だけで手元に揃う。

## 中心となる仕組み

git の refs は、ブランチやタグ以外にも任意の名前空間を持てる。
githive は `refs/projects/` 配下に機能ごとの ref を作り、各 ref を追記専用のコミットチェーンとして使う。

- コミットメッセージには機械可読なイベント（JSON）を格納する。
- コミットのツリーには人間可読な現在状態（markdown と JSON ファイル）を格納する。

この二重化により、`git log` でイベント履歴が、`git show <ref>:<path>` で現在状態が、専用ツールなしで読める。
専用ツールはイベントログを直接パースし、状態を再計算せずスナップショットを読むことで高速に動く。

## 設計原則

- **git ネイティブ**：データはすべて git オブジェクト（commit、tree、blob）とテキストで表現する。外部データベースを持たない。
- **ツールなしで読める**：素の git コマンドだけで全データを閲覧できる。各機能仕様に「素の git での読み方」を明記する。
- **オフラインファースト**：すべての操作はローカルで完結し、同期は fetch/push に集約する。ネットワークなしで issue の作成もコメントもできる。
- **決定的実体化**：状態はイベント集合の純粋関数である。同じイベント集合からは、誰がいつ計算しても byte 単位で同じツリーが得られる。これが競合解決（[03](03-sync-and-concurrency.md)）の基盤になる。
- **人間と Agent の対等**：CLI は人間向け出力と `--json` 出力を等価に提供する。Agent 専用の裏口 API を作らない。
- **既存ホスティング第一**：GitHub 等に refs を push するだけで全機能の中核が動く。自作サーバーはそれを強制力（権限、通知配信）で格上げする追加レイヤとする。

## 非目標

- リアルタイムチャットは目指さない。chat は push/fetch 周期で同期する非同期掲示板である。
- GitHub の Pull Request フローの置き換えは目指さない。コードレビューは対象外とする（将来の検討課題）。
- 大容量バイナリの格納は目指さない。添付ファイルは自作サーバーの LFS 対応（[12](12-forge-server.md)）まで対象外とする。
- 数千人規模の組織運用は目指さない。想定規模は 1〜50 人程度のプロジェクトである。

## 動作モード

サーバー側フックが使えるかどうかで、機能の実現度が変わる。
githive は次の 2 モードを定義する。

| 機能 | hosted モード（GitHub 等） | forge モード（自作サーバー） |
|------|--------------------------|---------------------------|
| issue / task / chat / wiki の読み書き | 完全動作 | 完全動作 |
| notify の記録 | 完全動作（配信は fetch 時に検出） | push 時に配信（WebHook、メール等） |
| 権限制御 | 助言的（ローカル検証と署名検証のみ） | 強制（pre-receive で拒否） |
| 署名検証 | クライアント側で `githive verify` | サーバー側で push 時に強制 |
| LFS / CI | 対象外 | 対応 |

hosted モードでは、悪意ある書き込み自体は防げない。
代わりに全イベントを SSH 署名し、検証で不正を検出可能にする（[11](11-security.md)）。
refs を使ったシステムに対応するサービスが現れた場合は、そのサービスの拡張点（フック相当）に合わせて forge モード相当へ格上げできる構造にしておく。

## 既知の制約

設計上受け入れた制約を先に明示する。

- **hosted モードでは、リポジトリへの write 権限が無い人は参加できない**。issue へのコメントも push である以上、read 権限だけの外部貢献者（OSS の利用者等）は書き込めない。forge モードでは policy で「issue と chat だけ push 可」の役割を作れる。fork 側の refs から取り込む pull 型の貢献フローは将来課題とする。
- **書いた内容は事実上削除できない**。イベント履歴は全 clone に複製され、改変は検証違反になる。表示上の訂正（supersedes）はできるが、履歴からの抹消はできない。秘密情報を書いてしまった場合の対処はローテーション（無効化）である（[11](11-security.md)）。
- **同期はリアルタイムではない**。共有の粒度は push/fetch であり、秒単位の共同編集は提供しない。

## 想定ユースケース

- 開発者がリポジトリを clone し、`githive init` の後、オフラインの移動中に issue を読み、コメントを書き、帰宅後に `githive sync` で共有する。
- AI Agent が clone 直後に `githive issue list --json` で未解決 issue を取得し、担当 task のステータスを `doing` に更新してから実装を始める。
- Agent が作業完了時に notify へ完了イベントを追記し、人間が次の fetch で通知を受け取る。
- チームがホスティングを GitHub から自作 forge に移行しても、記憶は refs ごと `git push --mirror` で移る。
