# 11. セキュリティ

## 脅威モデル

想定する敵対行為と、それぞれへの対処を先に列挙する。

| 脅威 | hosted モード | forge モード |
|------|--------------|-------------|
| 他人になりすましたイベント作成 | 署名検証で検出（事後） | push 時に拒否 |
| 権限のない ref への書き込み | 検出のみ（policy は助言的） | pre-receive で拒否 |
| 履歴の改竄（force push で過去を消す） | 各クライアントの reflog と分岐検出で発覚 | force を policy で禁止 |
| 台帳の不正変更（自分に admin を付与） | admin 署名要求の検証で検出 | push 時に拒否 |
| 鍵の漏洩 | key_revoke で以後の署名を無効化 | 同左 + サーバーが即時拒否 |
| 未来時刻イベントによる LWW の占拠 | 検出して警告（[03](03-sync-and-concurrency.md)） | now + 5 分超の ts を push 時に拒否 |

hosted モードの限界は明確である。
ホスティングへの書き込み権を持つ者の不正は「防止」できず、「検出と否認不可能性」までを提供する。
防止が必要な運用は forge モードを使う。

## SSH 署名

git のコミット署名機構（`gpg.format ssh`）をそのまま使う（[ADR-0007](adr/0007-ssh-signing.md)）。

- 署名対象はコミットそのもの。イベント JSON はコミットメッセージに含まれるため、署名はイベントの完全性を含む。
- 検証は users 台帳から allowed_signers 形式（`<email> <keytype> <pub>`。principal には台帳の emails[] を使う）を生成して `git verify-commit` 相当を行う。
- 検証規則：
  1. コミットの署名が台帳のいずれかの鍵で有効であること。
  2. イベントの `actor`（email）が、その鍵を登録した台帳ユーザーの `emails[]` に含まれること。actor はコミッタの email とも一致すること。
  3. 署名時刻が鍵の revoked_at より前であること。
  4. registry ref への変更は、変更時点の台帳で `role:admin` を持つ actor によること。

`githive verify` はチェーンを遡って全コミットを検証し、違反を一覧で報告する（終了コード 4）。
マージコミット（event-union）はマージ実行者の署名を持てばよい。中身の正しさは決定的実体化の再計算で独立に検証できる。

## 信頼の根

registry チェーンの最初のコミットを信頼の根とする（詳細は [features/users.md](features/users.md)）。
`githive verify` は信頼の根のハッシュを表示し、`githive.trust.root` に固定値が設定されていれば一致を要求する。

registry が無関係な 2 つのルートを持つ場合（同時初期化、または攻撃）、自動マージは行わない。
pin が設定されていれば pin と一致する側を正とし、無ければ検証エラーとして人間の判断を求める。
ULID の早い側を自動採用する規則は採らない。ULID は自己申告の時刻を含むため、後から「より古い」ルートを偽造できてしまう。

## Agent の扱い

AI Agent は `kind: agent` の独立ユーザーとして登録し、人間と別の鍵を持たせる。

- Agent の鍵は権限を絞れる（policy で Agent 用グループの書き込み先を制限する等）。
- 監査時に「人間の判断」と「Agent の行動」をイベントログ上で区別できる。
- Agent が暴走した場合、その鍵だけを revoke すれば人間の作業は影響を受けない。

## 秘匿情報

**書き込まれた秘密は削除できない前提で運用する**。
イベント履歴は全 clone に複製され、履歴の書き換えは署名とチェーン整合を壊す。
issue コメントにトークンやパスワードを貼ってしまった場合の正しい対処は、その秘密自体のローテーション（無効化）であり、これは中央管理の issue トラッカーでも本来同じである（削除しても閲覧済みの可能性は消えない）。
CLI は書き込み本文に秘密らしきパターン（既知のトークン形式）を検出したら投稿前に警告する。

githive のデータはリポジトリを読める全員に見える。
アクセス制御は「書き込み」に対してのみ提供し、「読み取り」の制御はリポジトリ自体のアクセス権に委ねる。
issue 単位の閲覧制限（confidential issue）は提供しない。設計上、暗号化を持ち込まない限り不可能であり、v1 の範囲では複雑さが利益を上回る。
秘匿が必要な情報はそもそも書かない、を運用規範とする。

## 実装上の注意（実装者向け）

- 署名検証は system git（`git verify-commit`）に委譲し、暗号処理を自前実装しない。
- allowed_signers ファイルは一時ファイルとして都度生成し、検証後に削除する。台帳の状態と検証結果の対応をログに残す。
- イベント JSON のパースは厳格モード（未知の封筒バージョン拒否、サイズ上限 256 KiB、深さ制限）で行い、細工された入力での資源枯渇を防ぐ。
- ref 名、ユーザー名、ファイルパスはすべて許可文字の白リストで検証してから使う（パストラバーサル防止。特にツリー書き込み時）。
