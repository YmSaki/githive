# 03. 同期と競合解決

## 前提

書き込みはローカル ref の更新で完結する（[01](01-architecture.md)）。
同期はそれと独立した操作であり、`githive sync` が fetch と push をまとめて行う。
CLI の書き込み系コマンドは既定で書き込み後に sync を試み、`--no-sync` で抑止できる。

## sync のアルゴリズム

対象 ref ごとに次を実行する。

```
sync(ref):
  fetch origin +ref:refs/githive-remote/<ref>   # リモート追跡（衝突しない一時名前空間）
  if remote == local:            完了
  if local が remote の祖先:      local を remote に fast-forward（他者の更新の取り込み）
  if remote が local の祖先:      push（fast-forward push、通常はこれで成功）
  else:                          分岐 -> event-union マージ（後述） -> push
  push が non-fast-forward で拒否されたら fetch からやり直す（最大 N 回、既定 5）
```

- 再試行が尽きたら終了コード 3 で失敗し、ローカル状態は保持する。次回 sync で再開できる。
- 全 ref 一括の sync は、fetch を 1 回のワイルドカード fetch に、push を複数 refspec の 1 回の `git push` にまとめる。git は ref ごとに受理・拒否を返すため、拒否された ref だけを再試行する。
- push は system git を使うため、認証（ssh-agent、credential helper）は利用者の git 設定がそのまま効く。
- 通常の `git fetch` は追跡名前空間（`refs/githive-remote/*`）だけを進め、ローカルの `refs/projects/*` には触れない（[02](02-data-model.md) の追跡分離）。IDE の自動 fetch が走っても未 push のローカルチェーンは壊れない。

### クラッシュ安全性とローカル競合

- マージの計算は一時オブジェクトとして行い、ローカル ref の更新は最後の 1 回の compare-and-swap（`update-ref` の旧値検証つき更新）だけにする。途中でプロセスが落ちても ref は旧状態のまま残る。
- 同一マシンで複数の githive プロセス（例：人間の CLI と Agent の MCP）が同じ ref に書く場合も、この CAS が衝突を検出し、負けた側は読み直して再適用する。

### 共通祖先が無い場合

2 者が独立に同じ ref を作った場合（同時 `githive init` による meta/config、registry 等）、チェーンは無関係な 2 つのルートを持つ。
event-union マージは空状態を共通祖先とみなし、通常どおり和集合で合流する。
例外は users/registry で、信頼の根が 2 つある状態は自動解決しない。`githive.trust.root` の pin と一致する側を正とし、pin が無ければ検証エラーとして人間の判断を求める（[11](11-security.md)）。

### 時計異常への防御

端末の時計が未来にずれていると、そのイベントはローカル時刻が追いつくまで LWW で勝ち続ける。
これを抑えるため次を規定する。

- クライアントはイベント作成時に、追跡している最新リモートイベントの ts より 5 分以上過去へ遡る自時計を検出したら警告する。
- sync とマージは、now + 5 分を超える ts のイベントを警告付きで受け入れる（hosted モードでは拒否できないため検出に留める）。
- forge は now + 5 分を超える ts のイベントを含む push を拒否する（[12](12-forge-server.md)）。
- ULID 生成器はプロセス内で単調性を保証する（同一ミリ秒内の連番、時計巻き戻り時は前回値+1）。

## event-union マージ

分岐した 2 つのチェーンは、次の規則で合流する（[ADR-0004](adr/0004-merge-by-event-union.md)）。

1. 両側のチェーンからイベント集合を収集する（共通祖先までで打ち切る）。
2. 和集合を取り、イベント ID の辞書順に並べる。同一 ID は 1 つに数える。
3. 共通祖先の状態を初期値に fold し、実体化ツリーを得る。
4. 2 親のマージコミット（message = `merge: event-union`、tree = 手順 3 の結果）を作る。

イベントは不変の事実であり、状態は fold の結果にすぎない。
そのため「同じイベント集合 → 同じ状態」という決定性（[02](02-data-model.md)）から、どちら側がマージしても同一ツリーに収束する。
CRDT の用語では、イベント集合を grow-only set とする operation-based な収束である。

**手動での競合解決は存在しない**。
同一フィールドへの同時変更は、イベント順序（ULID 順）で後のものが勝つ。
「勝った」変更も負けた変更もイベントログには両方残るため、履歴から意図の衝突を検出して人間が追加イベントで訂正できる。

## チェックポイント

チェーンが長くなると、fold の起点を毎回ルートまで遡るのは遅い。
これを避けるため、**チェックポイントコミット**を導入する。

- kind は `<機能>.checkpoint`。data に「この時点の完全な状態」と「包含するイベント数とハッシュ」を持つ。
- チェックポイント以降のイベントだけ fold すれば現在状態を再構成できる。
- 作成条件：直前のチェックポイントから 500 イベント、または 90 日経過。`githive fsck --compact` が作成する。
- チェックポイントは履歴を削除しない。単なる読み出しの近道である。

**チェックポイントは fold の対象ではない**。
和集合マージと状態再計算では常に無視し、線形履歴の読み出し高速化（自分より前のイベントがすべてチェーン上の祖先である場合の起点）にのみ使う。
この規則が無いと、次のデータ消失が起きる：A がオフラインで作った古い ULID のイベント群と、B が作ったチェックポイントをマージするとき、和集合の ULID 順では A のイベントがチェックポイントより前に並び、チェックポイントを状態として適用すると A のイベントが実体化結果から消える。
無視規則の下では、マージ時は共通祖先起点の完全な fold に落ち、正しさが保たれる。

## shallow / partial clone との関係

- `--depth 1` の fetch でも、先頭コミットのツリーに現在状態が完全に含まれるため、一覧と閲覧は動く。
- 履歴表示、検証（署名、チェーン整合）、分岐マージは深い履歴を要求する。必要時に `git fetch --deepen` で追加取得する。
- ツールは「浅い履歴しかない」状態を検出し、自動 deepen するか、明示的なエラーで知らせる。

## notify ストリームの競合特性

`refs/projects/notify/stream` は全員が書く単一 ref のため、non-fast-forward が最も起きやすい。
event-union マージは通知の性質（追記のみ、順序の厳密性が不要）と相性がよく、機械的に解決できる。
それでも push 競合の再試行が目立つ規模になったら、月次シャード（`refs/projects/notify/2026-07`）へ移行する。
シャード名の解決規則は [features/notify.md](features/notify.md) に定義済みで、移行はデータ形式の変更を伴わない。

## リモート巻き戻りからの自己修復

リモートが force push 等で過去に巻き戻された場合、sync は「local が remote より進んでいる」と判定してそのまま push し、履歴を自動復元する。
巻き戻しが分岐を伴っていても、event-union マージにより双方のイベントが保全される。
つまり hosted モードで ref を破壊されても、最新状態を持つ clone が 1 つでもあれば次の sync で治る。
この性質は攻撃対策というより、操作ミス（誤った force push、リポジトリ移行の失敗）からの回復手段として重要である。

## リモート追跡と GC

- リモート状態は `refs/githive-remote/*` に保持し、ローカルの `refs/projects/*` と分離する。
- すべての履歴はいずれかの ref から到達可能なので、git の GC で消えることはない。`core.logAllRefUpdates = always`（[02](02-data-model.md)）により、ref の付け替え事故も reflog から復旧できる。
- エンティティの「削除」は物理削除ではなく、`*.status = archived` 等の状態遷移で表す。ref の物理削除は forge モードの管理操作としてのみ許す。

## 既存ホスティングでの注意

- GitHub / GitLab は任意の refs 名前空間の push を受け付ける（`refs/pull/*` 等の予約名前空間を除く）。`refs/projects/*` は安全に使える。
- ホスティングの Web UI にはこれらの ref は表示されない。閲覧はローカルで行う。
- サーバー側フックが使えないため、push の直列化や検証はできない。整合性はこの章のクライアント側マージと、署名検証（[11](11-security.md)）で担保する。
