# ADR-0010: ライセンスは MIT + forge のみ AGPL とする

- 状態：採用
- 日付：2026-07-05

## 文脈

v0.1 公開に向けてライセンスを決める必要がある。
要求は次の 3 つである。

1. 仕組み（データ形式、`.workflows/` スキーマ、golden ベクタ）は誰でも使え、互換実装を歓迎する。plugin と標準化の賭けは採用の摩擦が少ないほど有利。
2. forge の実装だけは、フォークがプロプライエタリにクローズされない形にしたい。
3. 運用が面倒にならないこと。

## 決定

- 本リポジトリ（core、CLI、TUI、MCP、クライアント、spec、docs）は **MIT**。
- forge は実装時に**別リポジトリ**とし（[13](../13-roadmap.md) Horizon 4 の別リポジトリ化と同時に決着）、**AGPL-3.0-or-later** とする。

## 理由

- MIT の core を AGPL の forge が利用するのは適法（寛容 → コピーレフトの一方通行）。逆は不可なので、「core は server を import しない」という既存の依存規則がそのままライセンス境界の防壁になる。core への変更は必ず MIT 側リポジトリで行う。
- AGPL は forge のサービス提供者に改変ソースの公開を義務づけるが、商用ホスティングや自社利用は妨げない。要求 2 を満たしつつ、MIT の仕様から他者が独自 forge を書く自由（要求 1）を残す。
- 同領域の前例として Gitea（MIT）と Forgejo（GPL 系）が両立しており、この分割は実務上機能する。

## 帰結

- AGPL を一律回避する組織は forge の採用をためらう可能性がある。それらの組織は hosted モードか互換実装を使えばよく、githive のデータは失われない（forge はレイヤにすぎない）。
- forge リポジトリへのコントリビューションは AGPL で受け入れる。core に触れる変更は MIT 側へ誘導する。
- 本 ADR は法的助言ではなく方針の記録である。公開前に必要なら専門家の確認を挟む。
