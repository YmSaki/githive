# ADR-0001: git refs をデータストアにする

- 状態：採用
- 日付：2026-07-04

## 文脈

プロジェクト記憶（issue、task、notify、users、chat、wiki）を、clone するだけで人間と AI Agent の双方が読める形で保存したい。
候補は、

- (a) リポジトリ内の通常ファイル（ブランチにコミット）
- (b) 外部データベース + API
- (c) git refs 配下の専用名前空間

であった。

## 決定

(c) を採用する。
`refs/projects/*` 配下に機能ごとの ref を作り、git オブジェクトだけでデータを表現する。

## 理由

- (a) は記憶の更新がコード履歴を汚し、ブランチ運用（PR、rebase）と衝突する。issue へのコメントのたびに main が進むのは受け入れられない。
- (b) は「clone した瞬間に読める」という要件そのものを満たさない。
- (c) はコード履歴から分離されつつ、git の転送・署名・GC 到達性・履歴の全機構を継承する。既存ホスティングにもそのまま push できる。

## 帰結

- clone 既定では refs/projects/* が fetch されないため、`githive init` による refspec 追加（または手動 1 コマンド）が必要になる。この一手間はドキュメントで明示する。
- ホスティングの Web UI にデータが表示されない。閲覧はローカルのツール（CLI/TUI/拡張）が担う。
- git の転送効率とオブジェクト共有により、記憶の同期コストは通常の fetch/push と同程度に収まる。
