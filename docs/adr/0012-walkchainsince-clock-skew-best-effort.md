# ADR-0012: WalkChainSince の `--since` 早期終了はクロックスキュー下でベストエフォートとする

- 状態：採用
- 日付：2026-07-12

## 文脈

issue #22 は `githive log --since` が常に対象 ref の全履歴を `WalkChain` でデコードしてから捨てる非効率を指摘し、`internal/core/chain.WalkChainSince`（既存の `WalkChain` は無変更のまま新設）で早期終了を導入した（PR #45）。

早期終了の枝刈り条件は「非merge（単一親）区間では、エンティティのチェーンは追記専用なので親は常に子以下の ts を持つ」という前提に立ち、ある commit の envelope の `ts < since` を観測した時点でその親を辿るのをやめる、というものだった。

PR #45 の auto-pr-review が、この前提が [03-sync-and-concurrency.md](../03-sync-and-concurrency.md)「時計異常への防御」と矛盾することを指摘した。同節は次を明記している。

> 端末の時計が未来にずれていると、そのイベントはローカル時刻が追いつくまで LWW で勝ち続ける。（…）クライアントはイベント作成時に、追跡している最新リモートイベントの ts より 5 分以上過去へ遡る自時計を検出したら警告する。sync とマージは、now + 5 分を超える ts のイベントを警告付きで受け入れる（hosted モードでは拒否できないため検出に留める）。

つまり時計異常は**警告のみで拒否されない**。端末の時計が実際に巻き戻っていた場合、同一の単一親チェーン内でも親→子で `ts` が逆転しうる。この場合 `WalkChainSince` は子の `ts < since` を見た時点で親を辿るのをやめるため、本来 `since` 以降であるはずの祖先イベントをエラーなく静かに欠落させる（再現テストで確認済み。マージコミット直前で打ち切る問題ではなく、単一親区間のこの前提自体の問題）。

対応の選択肢は次の3つを検討した。

1. **ベストエフォートとして受容し、既知の限界として文書化する。**
2. 早期終了の最適化自体を見送り、`WalkChain` の全走査に戻す（issue #22 を wontfix）。
3. 枝刈り条件を「単一 commit の ts」ではなく「経路上で観測した累積最小 ts」に変えて再実装する。ただし時計異常自体は双方向・無制限（クライアント側は警告するのみで拒否しない）であるため、この変更をもってしても理論上の正しさは得られない（累積最小値も同じ前提の上に成り立つため）。実装・テストのやり直しコストに対して得られる保証の向上が小さい。

## 決定

**選択肢1を採用する。**

- `WalkChainSince` の実装（`internal/core/chain/chain.go`）はそのまま維持する。
- 非merge区間の早期終了は「通常のクロック挙動下では正しい」ベストエフォートの最適化であり、極端な時計異常（端末の時計が実際に巻き戻った状態でイベントを追記した場合）が起きたときに限り、`since` 以降であるはずの祖先イベントを取りこぼしうることを、`WalkChainSince` のドキュメントコメントと [10-cli-spec.md](../10-cli-spec.md) の `--since` の説明に明記する。
- 本プロジェクトは現時点でソロ運用が前提であり、悪意ある入力ではなく自分自身の端末の時計異常のみが原因になる、発生頻度の低いケースである。厳密な正しさより実装・保守コストの低さを優先する。

## 帰結

- `githive log --since` は「通常のクロック挙動下では常に正しい、高速なフィルタ」という契約になる。極端な時計異常時の完全性は保証しない（決定的実体化・fold意味論には無関係——`githive log` は materialize を経由しないため、CLAUDE.md の決定性不変条件そのものには抵触しない）。
- 将来、複数拠点・複数運用者を前提にする場合や、時計異常の発生報告が実際にあった場合は、選択肢3（またはより強い保証を持つ設計）を再検討する。その際は本 ADR を参照し、なぜ当初ベストエフォートで妥協したかを追える状態にしておく。
