# ADR-0004: 競合は event-union マージで自動解決する

- 状態：採用
- 日付：2026-07-04

## 文脈

複数のクライアントが同じ ref をオフラインで進めると、push 時に non-fast-forward が発生する。
人手の競合解決を要求すると、Agent の自律動作と通知・chat の実用性が損なわれる。

## 決定

分岐したチェーンは、イベントの和集合を ULID 辞書順で fold し直した 2 親マージコミットで合流する。
手動解決は存在しない。
順序は ULID（時刻 + ランダム）の全順序で決まり、スカラー値は後勝ち、集合は加除の適用、追記データは単純合併とする。

## 理由

- イベントを「不変の事実の集合」、状態を「集合の純粋関数」と定義すれば、和集合 + 決定的 fold は必ず収束する（grow-only set の CRDT と同型）。
- git のマージ機構（2 親コミット）をそのまま履歴表現に使えるため、素の git から見ても正当な履歴に見える。

## 帰結

- 同一フィールドへの同時変更は黙って後勝ちになる。意図の衝突は履歴に残るので、事後に人間が訂正イベントを積む。この「気付ける」ことを、verify や CLI の表示（マージで負けた変更の警告）で支援する。
- 端末時計のずれは順序に影響するが、全員が同じ順序を再現するので収束は壊れない。
- ULID の順序は因果関係を保証しない（返信が投稿より先の ID を持つことが理論上ありうる）。表示側は reply_to による並べ替えで補正する。

## 退けた代替案

- **hybrid logical clock（HLC）**：因果順序の精度は上がるが、封筒とマージ規則が複雑になる。想定規模（〜50 人、push 周期の同期）では時計ずれの実害が小さく、複雑さに見合わない。
- **3-way テキストマージ**：JSON の構造マージは意味を保証できず、自動収束が壊れる。
- **サーバーでの直列化**：hosted モード（サーバーなし）で成立しない。
