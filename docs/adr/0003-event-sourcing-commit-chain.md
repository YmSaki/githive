# ADR-0003: 1 コミット = 1 イベント、ツリー = 実体化状態の二重構造

- 状態：採用
- 日付：2026-07-04

## 文脈

各 ref にデータをどう格納するかを決める必要がある。
要件は、専用ツールなしの可読性、追記の履歴性、競合の機械的解決、の 3 つである。

## 決定

wiki を除く全 ref を追記専用のイベントチェーンとする。

- コミットメッセージ：要約 1 行 + イベント JSON（機械可読の正）。
- コミットツリー：そのイベントまでを fold した現在状態（人間可読のビュー）。

## 理由

- メッセージ側だけに置く案（git-appraise 方式）は、現在状態を得るのに全履歴の再生が必要で、「`git show` で今の状態が読める」を満たさない。
- ツリー側だけに置く案（状態のスナップショットのみ）は、変更の意図（イベント）が失われ、競合解決が 3-way テキストマージに退化して自動収束できない。
- 二重構造なら、`git log` = 履歴、`git show ref:path` = 現在、shallow fetch でも現在状態が完全、という 3 つの性質が同時に成り立つ。

## 帰結

- データは冗長になるが、状態ファイルの大半は変更されないため blob は共有され、実増分は小さい。
- ツリーとイベントの不整合（手書きミス、改竄）がありうるため、fsck / verify / pre-receive が「イベントからの再計算と一致するか」を検証する。
- wiki は例外的に通常ブランチ運用とする。理由は [features/wiki.md](../features/wiki.md) に記載。
